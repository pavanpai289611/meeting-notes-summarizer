# Retrospective — Meeting Notes Summarizer

**How the plan changed from the original.** The build followed the course's 8-phase structure throughout, but the depth of iteration within each phase was deliberately scaled back from what the doc recommends — one PRD pass rather than two-plus with gap/conflict checks, and a lighter Plan Mode ceremony for the Sprint 2 change request than the full workflow describes. What didn't scale back was verification rigor: every task was tested with real browser automation, real API calls, and deliberately forced failure states rather than accepted on code review alone. The most concrete plan change came from the Sprint 2 stakeholder change request itself — the "mobile-responsive layout" ask turned out to directly reverse an explicit scope decision already documented in the PRD's own out-of-scope list (§2.6, item 8). The original plan had no way to anticipate this, since it assumed the change request would be purely additive; instead, incorporating it meant updating a previously "settled" document, not just writing new code. A second, quieter plan shift: the assumption that most tasks would be build-then-verify turned out to undersell how much of the actual work was verification itself — several features that looked complete after implementation (the "Unassigned" owner-attribution fallback, the 20,000-character input path, the framework-level error handling in the security review) only became genuinely confirmed once deliberately tested with real, edge-case data rather than trusted from code review.

---

**The hardest part.** Verifying claims about the app's behavior with real evidence rather than reasonable-sounding assumptions. Several pieces of the app "looked correct by code review" for a long stretch of the build — the "Unassigned" owner-attribution fallback, the 20,000-character input handling — simply because every real test transcript used up to that point happened not to exercise those specific paths. Nothing was actually wrong, but the gap between "the code should handle this" and "this has been observed to handle this" was real, and closing it required deliberately constructing test cases (a transcript with no named owner, a 20K+ character paste with a truncation-detection marker) rather than trusting the first thing that worked.

---

**A real failure and how it was handled.** The security review surfaced a genuine gap: malformed or oversized requests never reached the app's own error handling at all — they fell through to Express's default behavior, leaking stack traces and absolute filesystem paths. This hadn't shown up anywhere earlier because every request sent during development happened to be well-formed. It was fixed with a global error handler mapping those cases into the same clean error envelope used everywhere else, then re-verified against the exact requests that originally exposed it.

---

**What I'd do differently with more time.** Test at the framework/middleware boundary earlier, not just the application-code boundary — the three documented error codes felt like complete error coverage because they mapped cleanly onto PRD acceptance criteria, but the PRD's criteria were never written with framework-level failures in mind. I'd also add a React error boundary and a rapid-double-click test for the summarize button, both flagged during the backlog audit as thin-on-verification but left unaddressed. And I'd budget verification as its own explicit phase of work rather than an implicit tail-end of each task — it consistently took as much attention as the implementation itself, and treating it as a given rather than a planned activity understated how central it actually was to the outcome.

---

**What I'm proud of.** The discipline of treating "looks right" and "verified right" as genuinely different states, and acting on that distinction consistently — through Playwright-driven browser tests, forced real failure states instead of only happy-path checks, and a security review that found and fixed a real issue rather than rubber-stamping a clean bill of health.
